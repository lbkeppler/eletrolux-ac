# Generic Entity Generation — Hardening Plan

> **For agentic workers:** Execute task-by-task with review between tasks. Tests run in WSL only (`wsl -d FedoraLinux-43 -- bash -lc 'cd /mnt/c/Users/Lucas/Documents/GitHub/eletrolux-ac && ~/electrolux-venv/bin/python -m pytest <args>'`).

**Goal:** Make entity generation genuinely data-driven so the integration works for ANY Electrolux/Frigidaire AC — not just the example fixture — validated against two real appliance shapes.

**Why:** Real-account validation (Electrolux YI09F, `applianceType: "CA"`) revealed the example fixture is not representative. The current code hardcodes assumptions that break on the real device: a fixed `FAN_MODE_MAP` drops `TURBO`; swing only checks `verticalSwing` (real device uses `flapOscillate`); `displayLight` is assumed to be an enum switch but is numeric (0-100); the switch table is a fixed list of names. `GET /info` also times out intermittently and needs retry.

**Chosen approach (design workflow winner — Hybrid):** Climate stays bespoke (it needs domain logic for HA's climate enums, the executeCommand on/off protocol, and C/F handling) but derives fan/swing/mode fully from the appliance's own capabilities. Everything else (switch/select/number/read-sensor) is generated by one pure function `classify_capability(name, cap)` that routes each capability by structure.

## Global Constraints

- Tests run in WSL (HA imports `fcntl`). WSL venv `~/electrolux-venv` = Python 3.14.5 + HA 2026.7.2 + PHACC 0.13.346.
- Two fixtures must BOTH pass: `tests/fixtures/{appliances,info,state}.json` (Frigidaire, type AC) and `tests/fixtures/real_ac_{appliances,info,state}.json` (Electrolux YI09F, type CA).
- The classifier is a PURE function — unit-test it directly over both fixtures.
- Rule ordering: check `values` BEFORE `min/max` (so `flapPosition` — type=number but carrying `POSITION_*` values — routes to select, not number).
- A capability is a SWITCH only if `access == "readwrite"` AND its value-set is boolean-like (`{ON,OFF}` or `{0,1}`) AND it's not consumed by climate. `access == "read"` → sensor/binary_sensor, never a control.
- Tolerant coercion helper maps `DISPLAY_LIGHT_N`/`POSITION_N` ↔ int, passes real numerics through, returns None if uncoercible. Used by number/select/displayLight so the enum-vs-numeric API inconsistency never crashes.
- Don't stage `.env`/`.har`/`.venv`. Commit after each green task.

## Task G1: `capabilities.py` — the pure classifier + coercion helper

**Files:** Create `custom_components/electrolux_ac/capabilities.py`; Test `tests/test_capabilities.py`.

**Produces:**
- `EntityKind` enum: `SWITCH`, `SELECT`, `NUMBER`, `SENSOR`, `BINARY_SENSOR`, `CLIMATE`, `IGNORE`.
- `snake_case(name: str) -> str`.
- `coerce_value(val) -> int | float | None` — `"DISPLAY_LIGHT_3"`→3, `"POSITION_2"`→2, `5`→5, `"7"`→7, `"ON"`→None.
- `CLIMATE_CONSUMED: frozenset` = mode, fanSpeedSetting, verticalSwing, flapOscillate (swing keys), targetTemperatureC, targetTemperatureF, ambientTemperatureC, ambientTemperatureF, temperatureRepresentation, executeCommand, applianceState.
- `SWING_KEYS: tuple` = ("verticalSwing", "flapOscillate", "horizontalSwing").
- `classify_capability(name, cap) -> tuple[EntityKind, dict]` — returns the kind + a small spec dict (e.g. `{"values": [...], "min": .., "max": .., "step": .., "on": .., "off": ..}`). Routing rules (in order):
  1. name in CLIMATE_CONSUMED → `CLIMATE, {}` (climate owns it).
  2. `access == "constant"` or type == "alert" or name startswith "hCPN_" → `IGNORE`.
  3. `access == "write"` (executeCommand, cleanFilterAlertReset) → `IGNORE`.
  4. Has non-empty `values`:
     - boolean-like set ({ON,OFF} or {0,1} or type boolean) AND `readwrite` → `SWITCH` (on/off from the actual keys).
     - boolean-like set AND `read` → `BINARY_SENSOR`.
     - >2 values (multi-enum) AND `readwrite` → `SELECT` (options = lowercased keys).
     - any values AND `read` → `SENSOR` (enum sensor, options = lowercased keys).
  5. Numeric (`min`/`max` present, no values):
     - `readwrite` → `NUMBER` (min/max/step).
     - `read` → `SENSOR` (measurement, numeric).
  6. else → `IGNORE`.

- [ ] Write `tests/test_capabilities.py` asserting the EXACT `(EntityKind, spec)` for EVERY capability in BOTH `info.json` and `real_ac_info.json`, plus `coerce_value` both directions. Key assertions: Frigidaire `displayLight`→SWITCH (values DISPLAY_LIGHT_0/1), YI09F `displayLight`→NUMBER (min0/max100), YI09F `flapPosition`→SELECT (values-first), YI09F `cleanAirMode`→BINARY_SENSOR (read), Frigidaire `cleanAirMode`→SWITCH (readwrite), YI09F `sleepMode`/`comfortAir`/`autoSenseMode`/`batchSchedulerMode`→SWITCH, `alerts`/`hCPN_*`→IGNORE, `executeCommand`→IGNORE, `currentEnergyUsePercent`→SENSOR, `flapOscillate`→CLIMATE (it's the swing key).
